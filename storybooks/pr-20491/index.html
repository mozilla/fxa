Summary
fix(pair): simplify /pair and /connect_another_device bootstrap
Description
Both pages now ask Firefox via WebChannel for a signed-in Sync user. /pair
shows the choice screen if signed in; otherwise it requests fresh sync
OAuth params and hard-navigates to / with oauth_webchannel_v1.

Direct visits to /connect_another_device hard-navigate to /pair when the
browser is signed in *and* isEligibleForPairing() (legacy Sync-context
plus pairing-entrypoint gate). When signed in but not eligible, the page
renders. When not signed in, the same OAuth flow runs. WebChannel
timeouts reveal the page as a safety fallback. Restores the legacy
redirect_immediately + redirect_to handoff to /settings.

The connect_another_device entrypoint uses an underscore form so it
clears the CMS endpoint validator and the Index page no longer crashes
on the redirect target. buildSyncOAuthSearch also forwards keys_jwk from
the fxa_oauth_flow_begin response, otherwise the OAuth code comes back
keyless and Sync never enables after oauth_login.

Glean view events on /pair now wait until the bootstrap reveals the
page, and the CAD Sign in button flips canSignIn to false on a
WebChannel failure so the click stops being a silent no-op.

Adds buildSyncOAuthSearch helper, FxAOAuthFlowBeginResponse type, and
the OAuthFlowBegin functional-test channel. New unit and functional
tests cover the bootstrap branches; the pair_preferences functional
test captures the WebChannel dispatch via a page-side promise so it
survives Firefox chrome tearing down the test tab.
