Summary
fix(settings): surface OAuth client-info fetch failures as discrete errors
Description
Because:

* When the in-app fetch to /v1/oauth/client/:id fails, fxa-settings
  silently builds a "ghost" OAuth integration with all clientInfo fields
  undefined. isTrusted() returns false, scope sanitisation strips every
  scope a real RP typically requests, and the flow throws
  "Invalid OAuth parameter: scope" (errno 109).
* The resulting Sentry events look like RP misconfiguration but the real
  cause is the client-info lookup never returning — confirmed by the
  extras shape `clientInfo: { redirectUri: "" }` and `trusted: false`
  across ~80k production events (FXA-CONTENT-1HYP, FXA-CONTENT-1GTG,
  and related groupings).
* The React-side regression was introduced by the GraphQL-to-REST
  migration (FXA-12995): Apollo's useQuery propagated error state to
  consumers, but the hand-rolled fetch replacement dropped that signal
  and useIntegration proceeds as if the lookup succeeded.

This commit:

* Adds a clientInfoLoadFailed boolean to GenericIntegration so the
  client-info lookup outcome is explicit on the integration itself.
* Has IntegrationFactory.initClientInfo set the flag when a client_id
  was present in the URL but this.clientInfo is undefined — i.e., the
  lookup was expected to run and didn't produce a result.
* Short-circuits OAuthWebIntegration.getPermissions to throw
  SERVICE_UNAVAILABLE (errno 998) when the flag is set, bypassing the
  scope-sanitisation path and its misleading Sentry capture. The
  existing try/catch in AuthAndAccountSetupRoutes already renders
  OAuthDataError for this throw, so user-facing behaviour is unchanged.
* Captures the underlying fetch error to Sentry from useClientInfoState
  under the dedicated tag area: 'useClientInfoState.fetch', separating
  the real cause from the downstream symptom groupings.
* Adds unit coverage for the new flag, the early-return, and the Sentry
  capture on both network errors and non-OK HTTP responses.

Note: this does not change fetch credentials or headers. A separate
infrastructure track owns the Fastly/WAF bypass for
GET /v1/oauth/client/:id (see FXA-13618 description).
